# Knowledge Consolidation — From Resources to Frameworks

A periodic practice for converting a body of project documents (reports, fact sheets, planning artifacts, policy memos, weekly updates) into a small number of named, working operating frameworks plus the patterns that link them.

The point is not to summarize the corpus. The point is to surface the operating logic that's already implicit in the work, name it, and make it reusable for the next decision instead of being reinvented each time.

## When to run a consolidation

- After the corpus has accumulated enough to support genuine cross-source evidence (typical threshold: 20+ substantive documents across 3+ sub-domains).
- When you notice you're answering the same kind of question repeatedly and reaching the same conclusion — that's a signal a framework is hiding in the work.
- Before a major planning cycle (year 2 of a demonstration project, new sprint launch, partnership onboarding) where reusable frameworks would change how the planning gets done.

## What you're looking for

Working frameworks have four properties:

1. **Cross-source evidence.** A principle that only appears in one document is suggestive, not working. A principle that appears in 4–9 unrelated documents is working.
2. **Already in operation.** The frameworks come from observed practice, not aspiration. If the team isn't actually doing it, it's not a framework — it's an idea.
3. **Reusable in new contexts.** A working framework can be applied to a fresh decision and produce a useful default. If naming it doesn't change anything next time, it wasn't worth naming.
4. **Traceable.** Every principle links back to specific source documents that demonstrate the principle in practice.

## Steps

### 1. Inventory the corpus

List every substantive document by domain and date. Don't skim. The first read is for surfacing — what's in here, what isn't, what's missing.

### 2. Cluster by theme

Pass through the corpus tagging each document with the operational theme(s) it embodies (substrate vs intelligence; demonstration vs scale; per-student vs aggregate; etc.). The themes that show up across many unrelated documents are the framework candidates.

### 3. Draft frameworks

For each candidate, write:

- **Statement** — one sentence that states the principle.
- **Evidence** — 3–9 sources that independently demonstrate it.
- **Confidence** — High / Medium / Low based on cross-source coverage and explicit articulation.
- **Boundaries** — when the principle does NOT apply.
- **Use cases** — concrete situations where the framework changes the default action.

### 4. Promote / hold

A framework is **working** when there's strong cross-source evidence and at least one explicit articulation in an authoritative document. Otherwise it's **emerging** — keep the draft but don't promote until more evidence arrives.

Resist the urge to promote everything to "working" on the first run. Most consolidations should produce 3–5 working frameworks plus 2–4 emerging ones. If you're producing 12, you're describing surface-level themes, not operating logic.

### 5. Identify patterns

Patterns are recurring shapes that show up *across* frameworks — meta-observations about how the work itself organizes. Two examples from the CPL corpus:

- **Scattered-to-systematic** — valuable state lives in scattered/ephemeral locations and gets converted to durable, structured stores with ingestion workflows. This is the meta-pattern that generates infrastructure-first scaling.
- **AI as accelerant, faculty as validator** — every AI deployment produces first-draft outputs that faculty workgroups review, validate, and approve. AI is never terminal authority. This is the governance pattern that lets a faculty-driven accredited system adopt AI safely.

Naming these patterns matters because they generate the next framework before you have enough source documents to build it from scratch.

### 6. Mark sources as consolidated

Every source document that fed the consolidation gets a marker — frontmatter, an index entry, a tag — recording when it was last consolidated. This lets the next consolidation skip already-processed material and focus on new evidence.

### 7. Plan the next consolidation

A working framework predicts what evidence would refine, validate, or contradict it. Plan the next consolidation to look specifically for that evidence:

- "Run again after the demonstration project ships its scalable model — that's the first real test of `sprint-based-execution`."
- "Promote `friction-reduction` from emerging to working when any push-based agent actually deploys."

The aim is not a yearly summary ritual. It's a feedback loop where each consolidation refines the framework set the next round of work uses.

## Anti-patterns

- **Consolidating prematurely.** Three documents and a hunch is not a framework. Consolidations done too early produce confident-sounding frameworks that don't survive contact with the next quarter's work.
- **Aspirational frameworks.** "We should..." principles that no one is actually doing. These are commitments, not frameworks.
- **One-source frameworks.** A single authoritative document doesn't make a framework — it makes a doctrine. Frameworks need cross-source corroboration.
- **Framework inflation.** Producing 12 working frameworks per round so it looks comprehensive. Working frameworks are scarce by definition.
- **Skipping boundaries.** A framework without explicit boundaries gets misapplied to cases where it doesn't fit and erodes confidence in the whole set.

## What you get

A small number of named, traceable, currently-true operating principles that change how the next decision gets made — not a literature review.
